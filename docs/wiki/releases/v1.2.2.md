# RunTiyul v1.2.2

Release status: published 2026-07-22<br>
App version: `1.2.2+7`

RunTiyul v1.2.2 establishes a permanent Android signing identity so later
releases can install over it while preserving local data. It also reports the
installed version, recognizes later upgrades, and makes the release metadata
gate safe for this repository's CRLF line endings.

## Highlights

### Android update safety

- Replaced CI-generated debug signing with one protected 4096-bit RSA release
  key stored in GitHub Actions secrets.
- Pinned the public release-certificate SHA-256 fingerprint in the workflow.
- Release builds fail when signing credentials are missing or incomplete.
- CI rejects a non-increasing Android `versionCode` and verifies the APK's
  package ID, version name, version code, and signing certificate before upload.
- Version extraction strips CRLF carriage returns before strict Bash validation.

### Installed-version awareness

- About shows the installed app version and build number from Android/iOS
  package metadata.
- The app records its first tracked version locally and shows a one-time update
  confirmation after a later build is installed.
- Version recognition is offline and does not send device or location data.

## Important Android Install Note

Android APKs through `v1.2.0` were signed by different ephemeral debug keys.
Android therefore cannot install `v1.2.2` over those builds. Existing users must
uninstall the old app before installing `v1.2.2`; Android normally removes that
installation's routes, activities, settings, and downloaded maps during the
uninstall.

`v1.2.2` is the first published permanent-signing baseline. Releases after it
must retain the same application ID and certificate and use a higher build
number, allowing a normal in-place update. That two-release physical-device
upgrade still needs to be demonstrated after the next signed version exists.

## Known Limitations

- The permanent certificate and CI checks can be verified now, but an actual
  in-place upgrade requires this baseline and a later APK on a physical device.
- Existing debug-signed installations cannot preserve local data through the
  one-time transition to the permanent key.
- Production offline-map provider approval, iOS runtime verification, and the
  physical-device checks already tracked in the wiki remain unresolved.
